1. CSSの優先度

1. CSSの優先度

1-1. 基本的には後から書いた方が優先


同じセレクタの場合、後から書いた方が優先されるのが基本原則です。

id/classなどで指定した場合も同じです。

HTML
<h1>CSS勉強会</h1>
<h2 class='theme'>CSSの優先度</h2>
CSS
h1 { color: blue; }
h1 { color: red; } /* 優先 */

.theme { color: blue; }
.theme { color: red; } /* 優先 */

1. CSSの優先度

1-2. 詳細度


CSSのセレクタは要素名/id/class/擬似クラス/擬似要素など様々な書き方ができます。
詳細度とは、CSSの宣言が適用される際の重みでスコアが高いものが優先されます。
詳細度のスコアはセレクタの指定方法によって計算されます。
複数の宣言が同じ詳細度の場合は最後に宣言されたものが適用されます。

HTML
<h1 id='title' class='title'>CSS勉強会</h1>
CSS
* { color: pink; }                /* 詳細度:   0 */
h1 { color: yellow; }             /* 詳細度:   1 */
.title { color: white; }          /* 詳細度:  10 */
[class='title'] { color: black; } /* 詳細度:  10 */
#title { color: green; }          /* 詳細度: 100 */

1. CSSの優先度

1-3. 詳細度(インライン記述)


HTMLのstyle属性で書いたものはCSSファイル内での定義より優先されます。
混乱する元になるので、スタイルに関することはCSSファイルに集約することをお勧めします。

HTML
<h1 id='title' class='title' style='color: red;'>CSS勉強会</h1>
CSS
* { color: pink; }                /* 詳細度:   0 */
h1 { color: yellow; }             /* 詳細度:   1 */
.title { color: white; }          /* 詳細度:  10 */
[class='title'] { color: black; } /* 詳細度:  10 */
#title { color: green; }          /* 詳細度: 100 */

1. CSSの優先度

1-5. !important


値の横に!importantを付けると最優先されます。
これが多用されてしまうと、新しくルールを定義しても効くはずのスタイルが効かないなどの現象を招いてしまいます。
以下のCSSの例であれば、詳細度0のユニバーサルセレクタ(*)で定義したルールが適用されます。
CSSを大きく破綻させてしまう元となるため、基本的には避けたい記述です。
しかし、優先度最強なのを利用して意図的に使うこともあります。
思ったようにスタイルが効かなかったり、表示が崩れている時にはどのルールが影響しているかを手っ取り早くデバッグすることができます。

HTML
<h1 id='title' class='title' style='color: red;'>CSS勉強会</h1>
CSS
* { color: pink !important; }     /* 詳細度:   0 */
h1 { color: yellow; }             /* 詳細度:   1 */
.title { color: white; }          /* 詳細度:  10 */
[class='title'] { color: black; } /* 詳細度:  10 */
#title { color: green; }          /* 詳細度: 100 */

2. スタイルの上書き

意図しないスタイルが当たってしまい困ったことはありませんか？

2. スタイルの上書き

2-1. 上書きとは


スタイルのルールはCSSだけでなくHTMLやJavaScriptなどそれぞれで定義できます。
ルールは同じプロパティに対して何度も定義することはでき、最終的に優先度の高いルールが適用されます。
ルールを上書きしてしまうと、使われないルールの定義が残り続けて保守しにくくなったり、
意図しないスタイルを当ててしまうことがあるため注意しなければなりません。
また、定義済みのルールが影響して詳細度の関係上、上書きしたくてもできないというケースがあります。

2. スタイルの上書き

2-2. 詳細度と上書きの関係


以下の例では、sample2.cssでBOX1の文字色を変えるルールを書いていますが、
sample1.cssで定義した.boxのルールの方が詳細度が高いため後で読み込まれているsample2.cssのルールが効きません。

HTML
<link rel='stylesheet' type='text/css' href='sample1.css'>
<link rel='stylesheet' type='text/css' href='sample2.css'>
<div id='boxes'>
  <div class='box1 box'>BOX1</div>
  <div class='box2 box'>BOX2</div>
  <div class='box3 box'>BOX3</div>
</div>
CSS(sample1.css)
#boxes { color: green; }
#boxes div.box { color: red; }
CSS(sample2.css)
/* .box1の文字色を指定 */
#boxes > .box1 { color: blue; }

2. スタイルの上書き

2-3. CSSファイル以外でのスタイルの定義


CSS/HTML/JavaScriptそれぞれでスタイルを定義できますが、必要以上に散らすのは危険です。
以下は、スタイルに関する変更(文字色変更)をしようとCSSファイルに手を入れても効かないという例です。

HTML
<link rel='stylesheet' type='text/css' href='sample.css'>
<script src='http://semooh.jp/jquery/js/jquery.js'></script>
<script src='sample.js'></script>
<style>
.box { color: yellow; }
</style>
<div class='box'>BOX</div>
CSS(sample.css)
.box { color: blue; }
JavaScript(sample.js)
$(function() {
  $('.box').css('color', 'red');
});

3. 便利なJavaScriptですが・・・

3. 便利なJavaScriptですが・・・

3-1. JavaScriptでいろいろやり過ぎている


JavaScriptを使うと要素を追加/削除したり、スタイルを定義したりすることができます。
JavaScirptで様々なことができるので便利ですが、やり過ぎには注意です。
基本的には構造に関することはHTMLに、見た目に関することはCSSに、動きに関することはJavaScriptに書く方のをおすすめします。

HTML
<script src='http://semooh.jp/jquery/js/jquery.js'></script>
<script src='sample.js'></script>

<div id='boxes'></div>
JavaScript(sample.js)
$(function() {
  // 既存の要素の子要素を作成
  $('#boxes').append(
    $('<div>').addClass('box1').text('BOX1'),
    $('<div>').addClass('box2').text('BOX2'),
    $('<div>').addClass('box3').text('BOX3')
  );
  // 追加した要素の文字色を指定/要素を非表示に
  $('#boxes > .box1').css('color', 'blue');
  $('#boxes > .box1').hide();
  // クリックで要素を表示
  $('#boxes').click(function(){
    $('#boxes > .box1').show();
  });
});

3. 便利なJavaScriptですが・・・

3-1. JavaScriptでいろいろやり過ぎている(改善例)


構造に関することをHTML、見た目に関することをCSSに移しました。
このようにすることで要素を追加したり、スタイルを変更する場合でもどこを編集すれば良いか予測しやすくなります。

HTML
<script src='http://semooh.jp/jquery/js/jquery.js'></script>
<script src='sample.js'></script>
<link rel='stylesheet' type='text/css' href='sample.css'>
<div id='boxes'>
  <div class='box1'>BOX1</div>
  <div class='box2'>BOX2</div>
  <div class='box3'>BOX3</div>
</div>
CSS(sample.css)
#boxes > .box1 {
  color: blue;
  display: none;
}
JavaScript(sample.js)
$(function() {
  // 既存の要素の子要素を作成
  $('#boxes').append(
    $('<div>').addClass('box1').text('BOX1'),
    $('<div>').addClass('box2').text('BOX2'),
    $('<div>').addClass('box3').text('BOX3')
  );
  // 追加した要素の文字色を指定/要素を非表示に
  $('#boxes > .box1').css('color', 'blue');
  $('#boxes > .box1').hide();
  // クリックで要素を表示
  $('#boxes').click(function(){
    $('#boxes > .box1').show();
  });
});

4. パフォーマンスを考えたCSSの書き方

CSSのパフォーマンスを考えながら書いていますか？

4. パフォーマンスを考えたCSSの書き方

4-1. CSSのセレクタにidを使うと速い？


CSSのセレクタにidを使うとレンダリング速度が速くなるというのは知られていますが、
それはidをキーセレクタ(セレクタの右端)に使うときに限って言えることです。
CSSセレクタは右から左に検索されるため、以下のようにidを使うことでむしろ遅くなることもあります。
このように、十分に要素が特定できるセレクタの場合に親要素を指定したり、要素名を付けるとパフォーマンスの低下を招きます。

HTML
<div id='boxes'>
  <div class='box'>BOX1</div>
  <div class='box'>BOX2</div>
  <div class='box'>BOX3</div>
</div>
CSS
/*
1. クラスが"box"の要素を探す
2. その親要素のidが"boxes"であるか調べる(現在の構造であれば余計な処理)
*/
#boxes > .box { color: blue; }

/*
1. クラスが"box"の要素を探す
*/
.box { color: blue;}

4. パフォーマンスを考えたCSSの書き方

4-2. セレクタの指定方法


もし、HTMLの中に100個のli要素があった場合、まず全てのli要素を探します。
現在のHTMLのツリー構造をそのままli要素まで辿るように指定してみると、非効率な検索をしていることが分かります。
該当するli要素に 'navigation-list' というクラスを付けるとCSSのパフォーマンスを上がりますが、HTMLまで手を入れる必要があります。
idセレクタは、ページに1つしか存在しないため余計な条件は省くべきです。

HTML
<div id='wrapper'>
  <div id='header'>
    <div id='navigation'>
      <ul>
        <li>リスト1</li>
        <li>リスト2</li>
      </ul>
    </div>
  </div>
</div>
CSS
/*
1. li要素を探す(100個該当)
2. 次にツリー構造を辿り、#navigationを探す
3. 更に、#headerを探す
4. 更に、#wrapperを探す
*/
#wrapper #header #navigation li { color: blue; }

/*
1. クラスが"navigation-list"の要素を探す
*/
.navigation-list { color: blue;}

/* 余計な条件を削除したもの */
#navigation li { color: blue; }

4. パフォーマンスを考えたCSSの書き方

4-3. @importと<link>


ロードしているCSSの中に表示しているページでは使わないスタイルのルールまで書いていることはありませんか。
適用しないスタイルのルールをロードするとパフォーマンスに影響を与えるだけでなく、保守性も下がります。
共通部分を集約したCSSを読み込ませたり、特定の機能にだけ適用するスタイルは切り出すなどCSSの分割を考えることも必要です。
@importは分割したCSSファイルを読み込むことができる便利な機能です。
但し、CSSファイルの最上部に書かれていないと機能しません。
以下のようにどちらでもCSSファイルの読み込みはできますが、パフォーマンスに少し違いがあるため説明します。

HTML
  <link type="text/css" rel="stylesheet" href="sample.css">
  <link type="text/css" rel="stylesheet" href="sample2.css">
  <link type="text/css" rel="stylesheet" href="sample3.css">
CSS(sample.css)
  @import url('sample2.css');
  @import url('sample3.css');

4. パフォーマンスを考えたCSSの書き方

4-3. @importと<link>


CSS側に@importを使って書くと、「sample.css」が @import で書かれた「sample2.css」「sample3.css」を読み込みます。
ただ、並列にダウンロードが走らないため待ちができてロードが遅くなります。

CSS(sample.css)
@import url('sample2.css');
@import url('sample3.css');

HTML側に<link>タグを使って書くと、ファイルを同時に読み込みすることができます。
すなわち、最初のCSSファイルの読み込み完了を待たずに次のファイルをリクエストできるため、
回線をより効率よく使ってダウンロードができます。

HTML
<link type="text/css" rel="stylesheet" href="sample.css">
<link type="text/css" rel="stylesheet" href="sample2.css">
<link type="text/css" rel="stylesheet" href="sample3.css">

5. CSSLintの紹介

5-1. CSSLintとは？

https://github.com/CSSLint/csslint
その名の通りCSSのLinterで、CSSに問題点ないか自動でチェックしてくれるツールです。
CSSLintにはいくつかルールがあり「この書き方はちょっとイケてない・・・」という点を警告してくれます。
ルールについては CSSLintの全32個のルールをサンプルコードを書きながら解説しました に詳しく書いています。
ブラウザ/CLIのどちらも提供されていて、簡単に導入/実行できます。

ブラウザ版

http://csslint.net/

CLI版

$ npm install -g csslint # CSSLintのインストール
$ csslint sample.css     # CSSLintの実行
